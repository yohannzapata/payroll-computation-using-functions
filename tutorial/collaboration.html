Collaboration Guide

Work together safely: make a branch, open a pull request, keep in sync, resolve conflicts. Use the cheat sheet anytime.

Home Setup Guide

Steps

  • 1. Create a branch & first commit

    Why: A branch is your own copy. Don't commit to main.

    Do this flow: get latest main → make a branch → edit → save with a commit → push:

    Get latest main

    git checkout mainGo to the main project. git pull origin mainGet the newest files from the team.

    Create a branch

    git checkout -b feature/yourname-taskMake your own copy to work on.

    See what's changed

    git statusShow what you changed.

    Stage and commit

    git add .Get your changes ready to save. git commit -m "feat: add basic tax calculation"Save your work with a note.

    Push your branch

    git push -u origin feature/yourname-taskSend your work to the team online.

    After pushing, click "Compare & pull request" on GitHub to propose your changes.

  • 2. Open a Pull Request

    Why: A PR is a request for review before merging to main.

    In GitHub: open the repo → “Compare & pull request” (or Pull requests → New). Use a clear title and a short description like:

    Implements basic tax rate (10%) applied to gross salary.
Updated computeNetPay() and added helper computeTax().
Test: Run program; enter sample salary 1000 → tax 100, net 900.
No external dependencies added.

    Add reviewers, then Create pull request. If someone requests changes, commit and push—your PR updates automatically.

  • 3. Sync with main

    Why: Main changes while you work; syncing avoids conflicts.

    Do this before pushing and at the end of the day:

    git checkout main
git pull origin main
git checkout feature/yourname-task
git merge main

    If conflicts appear, use the conflict steps below.

  • 4. Merge vs rebase

    Merge = simple history. Rebase = tidy history (advanced).

    Merge main into your branch (safe)

    git checkout feature/yourname-task
git merge main

    Rebase (optional advanced)

    git checkout feature/yourname-task
git rebase main

    If conflicts: fix files → git add fixed files → git rebase --continue

    Never rebase the shared main branch.

  • 5. Resolve merge conflicts

    Conflict: Two edits on same lines; choose or combine.

    See conflicts

    git status

    Open files in VS Code and choose Incoming/Current or edit manually:

    <<<<<<< HEAD
Your branch version
=======
Incoming (main) version
>>>>>>> main

    Stage resolved files

    git add path/to/file

    Finish merge

    git commit

    Or continue a rebase

    git rebase --continue

    Finally push

    git push

    Avoid conflicts: Pull often; keep branches small (1 feature); communicate with teammates about files you’ll modify.

  • 6. After merge

    Goal: Update local main and remove old branch.

    1. Switch to main and pull latest (your PR + others):
    git checkout main
git pull origin main
    1. Optionally delete your branch locally and remotely:
    git branch -d feature/yourname-task
git push origin --delete feature/yourname-task

    Create a new branch for the next change—never recycle old ones.

  • 7. Common errors
    • fatal: not a git repository – You’re in the wrong folder. Go to the project root.
    • authentication failed – Wrong credentials or expired token. Generate a new personal access token (PAT) and use it as password for HTTPS.
    • Updates were rejected because the remote contains work... – You didn’t pull changes before pushing. Run git pull --rebase origin main (on main) or merge main into your branch.
    • CONFLICT (content) – You have a merge conflict. Follow the conflict resolution steps above.
    • permission denied (publickey) – SSH key missing. Either set up SSH keys or use HTTPS clone.
Tip: When unsure, run git status. It tells you exactly what Git expects next. Head back to Setup if environment issues appear.
